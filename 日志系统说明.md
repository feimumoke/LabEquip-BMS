## 🎯 日志系统特性

### ✅ 核心功能

1. **文件输出**：日志自动写入文件,便于问题排查
2. **按天切割**：每天自动创建新的日志文件
3. **控制台同步**：开发环境可同时输出到控制台
4. **自动清理**：自动删除 7 天前的旧日志
5. **多级别支持**：DEBUG、INFO、WARN、ERROR、FATAL
6. **文件定位**：自动记录日志产生的文件名和行号
7. **线程安全**：支持并发写入
8. **优雅退出**：程序退出时自动关闭日志文件

## ⚙️ 配置说明

### 配置文件位置

`server/_config/conf.yaml`

### 配置示例

```yaml
# 日志配置
log:
  dir: "./logs"           # 日志目录
  level: "INFO"          # 日志级别: DEBUG, INFO, WARN, ERROR, FATAL
  enableConsole: true    # 是否同时输出到控制台
  gormLogLevel: "INFO"   # GORM SQL 日志级别: SILENT, ERROR, WARN, INFO
```

### 配置项说明

| 配置项 | 类型 | 默认值 | 说明 |
|-------|------|--------|------|
| `dir` | string | `"./logs"` | 日志文件存储目录 |
| `level` | string | `"INFO"` | 应用日志级别 |
| `enableConsole` | bool | `true` | 是否输出到控制台 |
| `gormLogLevel` | string | `"INFO"` | GORM SQL 日志级别 |

### 日志级别说明

**应用日志级别**：

| 级别 | 说明 | 使用场景 |
|------|------|---------|
| **DEBUG** | 调试信息 | 开发环境,详细调试 |
| **INFO** | 一般信息 | 正常运行日志（推荐） |
| **WARN** | 警告信息 | 潜在问题提醒 |
| **ERROR** | 错误信息 | 错误但不影响运行 |
| **FATAL** | 致命错误 | 严重错误,程序退出 |

**GORM SQL 日志级别**：

| 级别 | 说明 | 输出内容 |
|------|------|---------|
| **SILENT** | 静默模式 | 不输出任何 SQL 日志 |
| **ERROR** | 仅错误 | 只输出 SQL 错误 |
| **WARN** | 警告及以上 | 输出慢查询和错误 |
| **INFO** | 所有信息 | 输出所有 SQL 语句（推荐开发环境） |

## 📂 日志文件说明

### 文件命名规则

日志文件按天切割,命名格式：`bms-YYYY-MM-DD.log`

示例：
```
logs/
├── bms-2026-01-15.log
├── bms-2026-01-16.log
└── bms-2026-01-17.log
```

### 自动切割机制

- **切割时间**：每天 00:00:00 自动切换到新文件
- **检查频率**：每分钟检查一次
- **无缝切换**：切换过程不会丢失日志

### 自动清理机制

- **保留时间**：7 天
- **清理时机**：每次日志切割时检查
- **清理对象**：修改时间超过 7 天的日志文件

```bash
tail -f logs/bms-$(date +%Y-%m-%d).log

tail -n 100 logs/bms-$(date +%Y-%m-%d).log

# 查看错误日志

# 查看警告和错误

# 统计错误数量
grep -c "\[ERROR\]" logs/bms-$(date +%Y-%m-%d).log

# 查看特定时间段的日志
grep "10:30:" logs/bms-$(date +%Y-%m-%d).log

# 搜索关键词
grep "database connection" logs/bms-$(date +%Y-%m-%d).log
```

### 日志监控脚本

```bash
#!/bin/bash
# monitor_errors.sh - 监控错误日志

LOG_FILE="logs/bms-$(date +%Y-%m-%d).log"

while true; do
    ERROR_COUNT=$(grep -c "\[ERROR\]" $LOG_FILE 2>/dev/null || echo 0)
    if [ $ERROR_COUNT -gt 10 ]; then
        echo "警告: 检测到 $ERROR_COUNT 个错误!"
        # 发送告警通知
    fi
    sleep 60
done
```

## 🔍 问题排查

### 问题1：日志文件未创建

**检查清单**：
```bash
# 1. 检查日志目录是否存在
ls -la logs/

# 2. 检查目录权限
ls -ld logs/

# 3. 检查配置文件
cat server/_config/conf.yaml | grep -A 3 "log:"

# 4. 检查程序是否有权限写入
touch logs/test.log
```

### 问题2：日志未输出到控制台

**解决方案**：
```yaml
# 修改配置文件
log:
  enableConsole: true  # 确保设置为 true
```

### 问题3：日志文件过大

**解决方案**：

1. 调整日志级别（从 DEBUG 改为 INFO）
2. 减少日志保留天数
3. 手动清理旧日志：
```bash
# 删除 7 天前的日志
find logs/ -name "bms-*.log" -mtime +7 -delete
```

### 问题4：Docker 容器中日志丢失

**检查清单**：
```bash
# 1. 检查 volume 挂载
docker-compose config | grep -A 5 "volumes:"

# 2. 检查容器内日志目录
docker-compose exec backend ls -la /root/logs/

# 3. 检查宿主机日志目录
ls -la logs/
```

## 📈 性能说明

### 性能特点

- **异步写入**：日志写入不阻塞主流程
- **批量刷新**：自动批量刷新到磁盘
- **低开销**：日志系统本身开销极小

### 性能建议

1. **生产环境使用 INFO 级别**：避免 DEBUG 日志过多
2. **关闭控制台输出**：减少 I/O 开销
3. **定期清理日志**：避免磁盘空间不足
4. **使用 SSD 存储**：提高日志写入速度

## 🎓 最佳实践

### 1. 日志级别使用

```go
// ✅ 正确使用
log.Debugf("Request params: %+v\n", params)        // 调试信息
log.Infof("User %s logged in\n", username)         // 重要操作

log.Errorf("Database error: %v\n", err)            // 错误信息
log.Fatalf("Cannot connect to database: %v\n", err) // 致命错误

// ❌ 错误使用
log.Infof("Loop iteration %d\n", i)  // 不要在循环中打印 INFO 日志
log.Errorf("User not found\n")       // 业务逻辑不是错误
```

### 2. 日志内容规范

```go
// ✅ 好的日志
log.Infof("User login success, user_id=%s, ip=%s\n", userID, ip)
log.Errorf("Failed to query database, table=%s, error=%v\n", table, err)

// ❌ 不好的日志
log.Infof("success\n")               // 信息不足
log.Errorf("error: %v %v %v\n", a, b, c)  // 格式混乱
```

### 3. 敏感信息处理

```go
// ❌ 不要记录敏感信息
log.Infof("User password: %s\n", password)
```
